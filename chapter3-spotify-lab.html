The central connection
As you work through each step, pay attention to how the same dataset powers both the recommendation model and the Wrapped-style summary. That connection is the central business lesson of the Spotify case — and the goal of this lab is to experience it firsthand.

What You Will Build

A simplified Spotify-style system that:

  • Stores simulated user listening history (streams, skips, replays)
  • Uses collaborative filtering to recommend songs based on similar users' behavior
  • Demonstrates the training-serving gap — and what happens when it occurs
  • Generates a Wrapped-style year-in-review summary for a single user
  • Is published as a case study website on GitHub Pages

1
Understand the Business Problem

Before writing any code, use Claude to deepen your understanding of recommendation systems. Tell Claude you are in MIS 432 at Western Washington University and ask for plain-language explanations.

Write and reflect:

  • What problem is Spotify solving with its recommendation system? Why does it matter for the business?
  • What is collaborative filtering, in your own words? How is it different from simply recommending the most popular songs to everyone?
  • Spotify collects implicit signals (skips, replays, session time). Why are these more useful than asking users to rate songs explicitly?
  • What is the training-serving gap? Why is it dangerous specifically because it fails silently?
2
Build a Listening History Dataset

Start by creating simulated behavioral data — the raw material that powers the entire system.

Claude prompt
I am building a simplified Spotify-style recommendation system in Python for a class project. Please write beginner-friendly Python code that creates a simulated listening history dataset for 10 users and 15 songs. Each row should represent one listening event and include: a user ID, a song name, an artist name, a genre, and an implicit engagement score (a number from 0 to 1, where 0 means skipped and 1 means replayed multiple times). Make sure some users share similar listening patterns so collaborative filtering will actually work. Store this in a pandas DataFrame. Give me the complete code in one block I can paste into Google Colab, then explain what the data represents and why implicit signals matter in a real recommendation system like Spotify's.
Write and reflect
What does each row in your dataset represent? What would the equivalent row look like in Spotify's real system — what additional data would it contain?
3
Build a Collaborative Filtering Recommender

Now build the recommendation model using collaborative filtering.

Claude prompt
Using the listening history dataset from the previous step, please add code that: 1. Creates a user-song matrix where each cell contains a user's engagement score for a song (0 if they haven't listened) 2. Calculates similarity between users based on their listening patterns using cosine similarity 3. For a target user (User 1), finds the most similar users 4. Recommends songs that similar users enjoyed but User 1 hasn't heard yet 5. Prints the top 5 recommendations with the song name, artist, genre, and predicted score Give me all code from the previous step plus this new code in one block. Explain each step in plain English, and specifically explain what cosine similarity measures and why it works for finding users with similar taste.
Write and reflect
Look at the recommendations your system produced. Do they make sense given the listening data? What is the system actually doing when it finds "similar users" — and why does that work as a proxy for good recommendations?
4
Recreate the Training-Serving Gap

This is the most important step in the lab. You are going to deliberately break your system in the same way Spotify's system broke — and observe the silent degradation.

Claude prompt
I want to demonstrate the training-serving gap that caused Spotify's podcast recommendation system to silently degrade for four months. Using all the code from my previous steps, please add a demonstration that: 1. Shows the "correct" recommendations when training and serving data are processed identically 2. Introduces a subtle preprocessing difference — for example, normalizing engagement scores differently at serving time than at training time 3. Shows how the recommendations change (subtly, not dramatically) when this gap exists 4. Adds a data validation check that would catch this problem — comparing the statistical distribution of features at training time vs. serving time Give me everything in one code block. Then explain: why is this type of failure more dangerous than a system crash? What should an ML engineer monitor to catch this before it affects users for four months?
Write and reflect
Did the recommendations change dramatically or subtly? Why does the subtlety make this failure mode so dangerous from a business perspective? Connect your answer to the Spotify case specifically.
5
Generate a Wrapped-Style Summary

Use the same behavioral dataset to generate a personalized year-in-review — demonstrating that Wrapped and the recommendation engine are powered by the same data.

Claude prompt
Using the same listening history dataset from my previous steps, please add code that generates a Spotify Wrapped-style summary for User 1. The summary should include: - Top 3 most-played songs (highest engagement scores) - Top artist (most songs with high engagement) - Top genre - Total listening sessions - A fun personalized sentence like "You were in the top X% of listeners for [artist]" Print this as a clean, readable summary. Then explain: this summary uses the exact same data as the recommendation engine. What does that tell us about Spotify's infrastructure — and why does it matter for business strategy?
Write and reflect
Spotify Wrapped generates 600 million social media posts for free. It costs Spotify nothing extra to produce because it runs on the same data pipeline as everything else. What does this tell you about the relationship between AI infrastructure investment and marketing strategy?
